# Образы для подов — два варианта (baked vs s3-loading)

Снимок на `2026-06-25`. Этот документ описывает **образ, который менеджер
запускает на подах**, и существующий выбор между двумя его вариантами. Сам
`runpod_manager.py` образ-агностичен: он лишь дёргает GraphQL-deploy с
`template_id`/`image` из `PRESET` и потом опрашивает поды по фиксированному
контракту портов (см. ниже). Поэтому любой из вариантов работает с менеджером
без правок кода — меняется только поведение старта пода.

Исходники образов лежат в **отдельных репозиториях** (не в этом репо):

| Вариант | Репозиторий (локально) | Docker-тег | Внутренняя документация |
|---------|------------------------|-----------|--------------------------|
| **baked** | `C:\dev\runpod-baked` | `wikiniki/comfy_gpu_baked:latest` (+ base `…_base:latest`) | `claude/README.md`, `PLAN.md`, `PLAYBOOK.md`, `CRASH_DEBUG.md`, `NODE_VERSIONS.md` |
| **s3-loading** | `C:\dev\runpod-s3-loading` | `wikiniki/comfy_gpu:latest` | `claude/README.md`, `claude/OPTIMISATIONS.md`, `PATCHES_AND_SETUP.md` |

> «s3» в названии репозитория = устойчивое внешнее хранилище RunPod (Network
> Volume, S3-совместимое), смонтированное в `/workspace`. Образ s3-loading
> держит на нём **всё ПО** (ComfyUI + venv + ноды), а не только данные.

## Главное различие — где живёт ПО

| | **baked** | **s3-loading** |
|---|-----------|----------------|
| ComfyUI core | **в образе** (`/opt/ComfyUI`) | на volume (`/workspace/runpod-slim/ComfyUI`) |
| venv (torch, ~520 пакетов) | **в образе** (`/opt/venv`) | на volume |
| custom nodes (~46) | **запечены в образ** | клонируются/живут на volume |
| `.pyc` bytecode | предкомпилирован в образ (`/opt/pycache`) | компилируется на месте |
| models / output / input / user | на volume (симлинки image → volume) | на volume |
| **Время старта** | **~73 сек** | **~13 мин** |
| **Установки нод между рестартами** | **сбрасываются** (живут в слое образа) | **сохраняются** (на volume) |

Это и есть фундаментальный компромисс:

- **baked** — программа на локальном диске контейнера → старт в ~10x быстрее
  (нет чтения venv и нод по сети). Цена: всё, что доустановлено в рантайме
  (новая нода через ComfyUI-Manager, ручной `pip install`), исчезает при
  пересоздании пода, потому что находится в эфемерном слое образа, а не на
  volume. Обновление состава нод = пересборка образа (`nodes.list`).
- **s3-loading** — ComfyUI/venv/ноды лежат на network volume → любые установки
  переживают рестарт пода. Цена: ~13 минут старта (чтение по сети + установка
  зависимостей + force-reinstall torch при добавлении нод).

Выбор под задачу: baked — для быстрых одноразовых подов со стабильным набором
нод; s3-loading — когда пользователь активно ставит/правит ноды и не хочет
терять их при рестарте.

## Контракт с менеджером (одинаков у обоих образов)

Менеджер не знает, какой именно образ запущен — он полагается на **общий
контракт**, который реализуют оба `start.sh`:

- **Порт 8188** — ComfyUI, `/system_stats` → `serviceReady`
  (см. `pod-lifecycle.md`, фаза READY).
- **Порт 8189** — лёгкий `python3 -m http.server` поверх `/tmp/boot/`,
  отдаёт `/status.json` (BOOT-прогресс) и `/runtime.json` (активность).
  Поднимается **первым делом** в обоих образах, до старта ComfyUI.
- Поля `status.json` (`stage`/`pct`/`msg`/`elapsed`) и `runtime.json`
  (`active`/`queue_depth`/`total_started`/`total_completed`/…) одинаковы.

Пока этот контракт держится, `list_pods()`, прогресс-бар и idle-таймер
менеджера работают независимо от выбора образа.

## ⚠️ Флаг #1 — расхождение имён образов

`PRESET` менеджера (`runpod_manager.py`, см. `architecture.md`) ссылается на:

```
image:       wikiniki/comfy_runpod:latest
template_id: i3j2sm66q8
```

Но реально собираемые образы называются **`wikiniki/comfy_gpu_baked:latest`**
и **`wikiniki/comfy_gpu:latest`** — строки `wikiniki/comfy_runpod:latest` среди
них нет. На деле под создаётся по `template_id`, и фактический образ определяется
тем, что прописано в RunPod-шаблоне на стороне RunPod — а его из этого репозитория
не видно.

**Per-project выбор образа (2026-06-25).** В админ-панели появился блок «🖼 Образы
подов»: каталог записей `{label, template_id}` и per-project назначение шаблона.
При каждом деплое `template_id` резолвится через `resolve_template_id(project)` и
подставляется в `DeployOnDemand`/CLI `--template-id`. Каталог засеян записью
`{label:"Текущий (comfy_runpod)", template_id:"i3j2sm66q8"}`, что сохраняет
текущее поведение до явного переключения. Подробности — `docs/admin-panel.md`,
`docs/graphql-deploy.md`.

`PRESET["image"]` при этом остаётся только CLI-fallback/display меткой и в
GraphQL-путь не передаётся (образ определяется шаблоном). Расхождение имени
`wikiniki/comfy_runpod:latest` с реальными тегами по-прежнему открыто: нужно
открыть шаблон `i3j2sm66q8` в RunPod UI, сверить фактический тег и привести
`PRESET["image"]` к правдивому значению.

## ⚠️ Флаг #2 — TODO #1 закрыт только в одном образе

`TODO.md` (TODO #1, «залип `загружен · N в очереди`») предписывает фикс
**только в `start.sh`**: stale-activity safety-net — если `total_started >
total_completed` и последнее событие старше ~30 минут, подтянуть
`total_completed = total_started`, чтобы под не висел вечно «занятым».

Состояние на `2026-06-25`:

- **s3-loading** — safety-net **реализован**. `claude/README.md` (раздел
  «Boot-сервер статуса») явно описывает третий watcher: «Stale activity safety
  net — раз в минуту проверяет `runtime.json`; если `active=true`, но событие
  старше 30 минут — прощает зависшие промпты».
- **baked** — safety-net **отсутствует**. Runtime-watcher в `start.sh` (секция
  4b) имеет только orphan-guard (`completed` не превышает `started` через
  условие `[ "$started" -gt "$completed" ]`), но **нет** периодической проверки
  возраста `last_event_at` и подтяжки счётчика. Значит на baked-подах баг
  залипшего idle-таймера по-прежнему возможен → idle-cleanup может не сработать
  → деньги текут.

**Что сделать:** перенести stale safety-net из `runpod-s3-loading/start.sh` в
`runpod-baked/start.sh` (тот же подход B из `TODO.md`). Менеджер при этом не
меняется — он продолжит читать `active` из `runtime.json`.

## Связь с предложением по PULL-прогрессу

`docs/pull-progress-proposal.md` описывает «слепую зону» до того, как ответит
порт 8189. Её природа различается по образам:

- **baked** — образ большой (в нём всё ПО), поэтому слепую зону занимает в
  основном **pull самого образа** RunPod-ом (до запуска `start.sh`). Вариант 4
  предложения (уменьшить размер образа) здесь особенно уместен.
- **s3-loading** — образ небольшой, но ~13 минут уходят на чтение/установку с
  volume **после** старта `start.sh`. Здесь 8189 поднимается рано, а «длинная»
  фаза — это уже BOOT/установка нод, которую частично видно через `status.json`.

Это стоит учитывать при реализации фазы PULLING: для s3-loading значительная
часть ожидания уже наблюдаема через 8189, а для baked — нет.

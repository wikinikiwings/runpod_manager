# Прогресс фазы PULL через лог-эндпоинт консоли RunPod — план исследования

> **Статус: ОТЛОЖЕНО (2026-06-26).** Задача не начата, ждёт ручного захвата
> через F12. Этот файл — самодостаточный план, чтобы возобновить с нуля.
> Связано: [pull-progress-proposal.md](pull-progress-proposal.md) (UI-варианты),
> [graphql-deploy.md](graphql-deploy.md) (прецедент реверса `DeployOnDemand`).

## Цель

Показать пользователю прогресс во время **развёртывания контейнера** (pull +
распаковка образа) — «слепой зоны» до того, как поднимется наш boot-сервер на
порту 8189. Сейчас менеджер ловит только момент, когда контейнер уже развёрнут
и стартует ComfyUI. Для baked-образа (всё ПО запечено → большой образ) именно
pull занимает основное время, а **уменьшить образ нельзя** (решение владельца),
поэтому остаётся попытаться получить сигнал прогресса извне.

## Почему «изнутри» нельзя

`start.sh` (и сервер на 8189) запускаются **внутри уже скачанного** контейнера.
Pull/распаковка идут до этого — наш код там ещё не выполняется. Значит
инструментировать pull можно только со стороны **API/консоли RunPod**.

## Что уже выяснено об API RunPod (ресёрч 2026-06-26)

- **Официального поля прогресса pull нет.** GraphQL/REST по поду отдают:
  `desiredStatus` (`CREATED`→`RUNNING`), `lastStatusChange`, `lastStartedAt`,
  `runtime` (null, пока под не запущен). Нет `imageDownloadProgress` /
  `provisioningStatus` / процента слоёв. (Стадии `PENDING/BUILDING/UPLOADING`
  есть только у `hubRelease` — это сборка Hub-образов, не наш pull.)
- **System logs RunPod содержат строки pull** («download, extraction, start,
  stop»), но **доступны только в веб-консоли**. Через документированное API
  логи пода получить нельзя — открытый feature request
  [runpod-python#400](https://github.com/runpod/runpod-python/issues/400),
  не реализован; публично эндпоинт консоли никто не реверсил.
- **Вывод:** настоящий % недоступен официально. Единственный шанс на «настоящие»
  строки pull — **реверс недокументированного лог-эндпоинта консоли** (тем же
  методом F12, которым нашли `DeployOnDemand`). Это хрупкий путь — берём только
  как попытку, не как опору.

Источники: [docs.runpod.io/pods/logs](https://docs.runpod.io/pods/logs),
[runpod-python#400](https://github.com/runpod/runpod-python/issues/400),
[graphql-spec.runpod.io](https://graphql-spec.runpod.io/).

## Критерии реализуемости (3 go/no-go вопроса)

Захват через F12 должен ответить на три вопроса:

1. **Транспорт/эндпоинт** — GraphQL-запрос на `api.runpod.io/graphql`,
   WebSocket (`wss://…`), или отдельный HTTP/SSE-стрим? Определяет, как
   воспроизвести у нас.
2. **Авторизация (ГЛАВНЫЙ вопрос)** — принимает ли эндпоинт тот же
   `Authorization: Bearer <RUNPOD_API_KEY>` (ключ формата `rpa_…`), что и
   `DeployOnDemand` → **воспроизводимо с бэкенда**. ЛИБО он требует
   короткоживущий консольный токен/cookie/JWT (`eyJ…`) из браузерного логина →
   **с бэкенда повторить нельзя, это дилбрейкер.**
3. **Контент в слепой зоне** — реально ли видны строки `Downloading`/
   `Extracting`, **пока под ещё разворачивается** (до 8189)? Если логи текут
   только после старта контейнера — для слепой зоны бесполезно.

## Протокол захвата через F12 (делает владелец — нужна его сессия runpod.io)

Ловить во время реального pull: поднять **новый** под (на «холодном» хосте
baked качается ощутимо) и **сразу** открыть Logs → вкладку **System**, пока
статус «Downloading».

1. F12 → **Network**, включить **Preserve log**.
2. Открыть Logs пода в консоли (во время Downloading).
3. Отфильтровать:
   - **WS** — если есть `wss://…`: открыть → вкладка **Messages** → скопировать
     3–5 фреймов (особенно с «Downloading»/«Extracting»/слоями).
   - **Fetch/XHR** — найти запрос, в ответе которого текст логов (искать по слову
     «Downloading»).
4. Снять с найденного запроса/сокета:
   - **Request URL** (полный) и **Method**;
   - **Request Headers** целиком, но значения `Authorization`/`Cookie`/токенов
     **замазать** — указать только тип/формат (`Bearer rpa_****` vs `Cookie:
     __session=eyJ****`);
   - **Payload** (для GraphQL — `operationName`, `query`, `variables`);
   - **кусок Response** (5–10 строк с download-логом).

### Шаблон для заполнения (секреты — звёздочками)

```
ТРАНСПОРТ: WS | Fetch/XHR | SSE
URL:
METHOD:
AUTH (тип/формат, значение замазать):
PAYLOAD (для GraphQL — operationName + query + variables):
RESPONSE (5-10 строк с download-логом):
ВИДНО "Downloading"/"Extracting" В СЛЕПОЙ ЗОНЕ: да/нет
```

## Дерево решений по результату захвата

- **Auth = `Bearer <API_KEY>` (`rpa_…`) + HTTP/GraphQL** → почти наверняка
  реализуемо. Дальше: написать диагностический `test_logs.py` (по образцу
  `test_graphql.py`/`test_deploy.py` в корне репо), вызвать эндпоинт с бэкенда
  нашим ключом, подтвердить, что строки pull приходят.
- **Auth = консольный JWT/cookie** → server-side не воспроизвести без
  браузерных кред пользователя → **вердикт: нельзя** (без хрупких костылей).
  Тогда откатываемся на [pull-progress-proposal.md](pull-progress-proposal.md)
  Вариант 1+2 (вычисляемая фаза PULLING + оценочный бар по времени, без
  зависимости от RunPod).
- **Транспорт = WebSocket** → реализуемо, но проект намеренно **stdlib+Flask без
  зависимостей**, а в stdlib нет WS-клиента. Решение отдельным шагом: либо
  raw-socket WS-handshake, либо добавить `websockets`/`websocket-client`
  (нарушает «no deps» — нужно явное согласие). HTTP/SSE-стрим воспроизводим на
  `urllib` без новых зависимостей.

## Эскиз реализации, если зелёный

1. `test_logs.py` (корень) — изолированно дёргает эндпоинт с `_api_key`, печатает
   ответ. Доказательство feasibility без правок бэкенда.
2. Парсер строк pull → процент/стадия (формат недокументирован → защищаться
   try/except, дефолт «без процента»).
3. Новый кэш+poll по образцу `check_pod_boot_status` (порт 8189): отдельный
   TTL-кэш, `ThreadPoolExecutor`, дёргается из `list_pods()` только для подов в
   слепой зоне (`desiredStatus`=CREATED/`runtime`=null/8189 молчит).
4. UI: ветка фазы PULLING в `FRONTEND_HTML` рядом с BOOT-прогрессом.
5. Везде — graceful degradation: если эндпоинт/формат изменился, тихо падаем на
   спиннер без процента (как и прочие health-чеки).

## Ограничения и решения

- Baked-образ **не уменьшаем** (решение владельца) — путь «сократить pull» закрыт.
- «No new deps» (stdlib+Flask) — учитывать при выборе транспорта (WS = проблема).
- Реверс недокументированного эндпоинта **хрупок** — обязательна graceful
  degradation и пометка в коде, что путь может отвалиться при изменениях RunPod
  (как уже отмечено для `DeployOnDemand` в [graphql-deploy.md](graphql-deploy.md)).

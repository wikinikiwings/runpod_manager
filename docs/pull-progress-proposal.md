# Прогресс-бар для фазы PULL (скачивание/разворачивание образа)

> Новый документ с предложениями. НИЧЕГО в существующем коде/файлах не меняет —
> только описывает варианты реализации для обсуждения.
>
> Контекст: текущий прогресс-бар питается от `/status.json` (порт 8189), который
> поднимает `start.sh` ВНУТРИ уже скачанного контейнера. Фаза «Downloading your
> container» (pull + распаковка образа, иногда ~10 мин) идёт ДО этого — порт 8189
> ещё не отвечает, и пользователь видит «чёрный ящик».

## Суть проблемы

```
create ──► [PULL: качается образ, ~10 мин] ──► start.sh ──► BOOT(8189) ──► READY(8188) ──► RUNNING
           ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
           слепая зона: 8189 ещё молчит, прогресса нет
```

Фундаментальное ограничение RunPod: **API не отдаёт процент скачивания образа.**
Ни GraphQL (`query pod`), ни REST `/v1/pods` не возвращают «N из M слоёв» или
проценты. Индикатор «48% layers» в дашборде RunPod — внутренний, наружу не
выведен. Значит «настоящий» прогресс pull построить нельзя; можно только дать
осмысленный индикатор ожидания.

## Текущие фазы (как их видит менеджер сейчас)

Из `list_pods()` (architecture.md, строки 1013–1222) для каждого пода уже
известно:
- `desiredStatus` (CREATED/RUNNING/EXITED...)
- `serviceReady` (8188 /system_stats отвечает)
- `bootStage`/`bootPct`/`bootElapsed` (8189 /status.json)
- `runtime` (8189 /runtime.json) или null

Сейчас UI различает: BOOT (есть bootStage) → READY (serviceReady) → RUNNING
(runtime активен). А промежуток «под создан, но 8189 ещё молчит» в отдельную
фазу не выделен — это и есть слепая зона PULL.

---

## Вариант 1 — выделить фазу PULLING (рекомендуется, минимальные данные)

**Идея:** ввести в UI новое вычисляемое состояние между «создан» и BOOT.

**Признак PULLING** (всё уже доступно в list_pods, новых источников НЕ нужно):
```
под присутствует в списке (desiredStatus есть)
  И serviceReady == false      (8188 молчит)
  И bootStage пуст/None         (8189 /status.json не отвечает)
  И runtime == null             (8189 /runtime.json не отвечает)
```
Это однозначно «образ ещё качается/разворачивается, контейнер не стартовал».

**UI:** спиннер + честный текст, например:
«Скачивание и распаковка образа… Первый запуск на новой машине может занять
до ~10 минут. Это происходит до старта ComfyUI, поэтому детального прогресса
здесь нет.»

**Плюсы:** ноль новых внешних запросов, ложится на существующую логику
(ещё одно вычисляемое состояние в обогащении пода). Различение фаз —
чистая интерпретация уже собираемых полей.

**Нюанс:** «8189 молчит» бывает и в краткий момент, когда контейнер уже
запущен, но сервер 8189 ещё не поднялся. Но `start.sh` поднимает 8189 ПЕРВЫМ
делом, так что это окно — секунды; 95%+ слепой зоны = именно pull. Отнести
это состояние к PULLING корректно.

---

## Вариант 2 — оценочный (псевдо) прогресс по времени

Поверх Варианта 1: раз реального процента нет, показывать ОЦЕНОЧНЫЙ бар по
`elapsed` от момента создания пода (`createdAt`/`lastStatusChange` из GraphQL).

```
pull_elapsed = now - pod_created_at
estimated_pct = min(95, pull_elapsed / EXPECTED_PULL_SECONDS * 100)
```
где `EXPECTED_PULL_SECONDS` — настраиваемая оценка (например, 600).

**UI:** бар + подпись «оценка (~X из ~10 мин)». Обязательно пометить как
оценку, не выдавать за реальный прогресс. Когда появляется bootStage (8189
ответил) — переключиться на настоящий прогресс BOOT.

**Плюсы:** пользователь видит движение все 10 минут.
**Минусы:** оценка может расходиться с реальностью (холодный/тёплый хост
RunPod кэширует слои — время pull нестабильно). Поэтому верхний предел 95%
и честная подпись важны.

## Вариант 3 — парсинг system logs пода (НЕ рекомендуется как основной)

RunPod отдаёт system logs (docs.runpod.io/pods/logs), где видны строки про
pull слоёв. Теоретически можно тянуть и показывать «слой N». Но формат
недокументирован и может меняться → хрупко. Подходит максимум как
необязательное дополнение, не как опора.

> Ресёрч 2026-06-26 уточнил: логи доступны **только в веб-консоли**, через
> документированное API их нет ([runpod-python#400](https://github.com/runpod/runpod-python/issues/400)).
> Единственный путь к «настоящим» строкам pull — реверс недокументированного
> лог-эндпоинта консоли через F12 (как нашли `DeployOnDemand`). План и протокол
> захвата — отдельный файл [pull-progress-log-tap-investigation.md](pull-progress-log-tap-investigation.md)
> (задача отложена).

## Вариант 4 — атаковать причину: уменьшить размер образа

Самый радикальный и устойчивый путь: чем меньше образ, тем короче pull и тем
меньше слепая зона. Кандидаты на чистку в baked-образе (из requirements.freeze):
- 4 версии OpenCV (opencv-python + contrib + обе headless) → обычно нужна одна;
- 2 версии onnxruntime (onnxruntime + onnxruntime-gpu);
- ревизия тяжёлых research-пакетов.
ВАЖНО: venv воспроизводился 1:1 намеренно («работает»), поэтому любая чистка —
отдельная задача с тестированием, не трогать без проверки воркфлоу.

Также помогает многослойность + стабильный тег: RunPod кэширует слои образа на
хостах, повторный pull того же тега в том же DC быстрее. Двухступенчатая сборка
(base + тонкий start.sh) уже снижает частоту полных «холодных» pull, т.к.
тяжёлые слои base не меняются при правке start.sh.

## Рекомендация

**Вариант 1 + Вариант 2 вместе:** выделить фазу PULLING по уже доступным
сигналам и показывать в ней оценочный бар по времени с честной подписью.
Это даёт обратную связь все ~10 минут, не требует недокументированных
источников и ложится на существующую архитектуру `list_pods()` без новых
внешних зависимостей. Параллельно — Вариант 4 как стратегическое снижение
самой длительности pull.

## Где это концептуально подключается (без правок, ориентир)

- Вычисление фазы PULLING — в обогащении пода внутри `list_pods()`
  (architecture.md: Pod listing, строки 1013–1222), рядом с тем, где уже
  считаются serviceReady/bootStage/runtime.
- Отрисовка — в `FRONTEND_HTML` (строки 2077–3262), там же, где рендерится
  текущий BOOT-прогресс-бар: добавить ветку для состояния PULLING.
- `EXPECTED_PULL_SECONDS` логично сделать настройкой (по аналогии с
  DEFAULT_SETTINGS, строки 80–103), чтобы подгонять под реальную статистику.

Никаких новых health-эндпоинтов и внешних вызовов Вариант 1+2 не требует:
всё считается из полей, которые list_pods() уже собирает.

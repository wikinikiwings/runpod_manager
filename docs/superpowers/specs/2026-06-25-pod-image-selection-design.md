# Design: выбор образа пода (пер-проект)

Дата: 2026-06-25
Статус: approved (brainstorming), ожидает плана реализации
Версия кода на момент дизайна: v6.6

## Проблема

Все поды запускаются на единственном захардкоженном образе: GraphQL-мутация
`DeployOnDemand` передаёт `templateId = PRESET["template_id"]` (`i3j2sm66q8`),
а имя Docker-образа берётся из самого RunPod-шаблона. У команды есть несколько
вариантов образа (например, `baked` — быстрый старт, и `s3-loading` — установки
нод переживают рестарт; см. `docs/pod-images.md`), но переключить образ можно
только правкой кода `PRESET`.

Нужно: в админ-панели дать возможность вести каталог образов и назначать образ
**каждому проекту**, чтобы поды проекта запускались на выбранном образе.

## Решения, принятые при брейнсторминге

1. **Гранулярность — пер-проект.** Админ назначает образ каждому проекту
   (по аналогии с `project_quotas`). Поды проекта запускаются на его образе.
2. **Каталог — админ-редактируемый.** Админ через «+» добавляет записи
   `{понятное название → template_id}`. В UI видно только название; при запуске
   подставляется `template_id`.
3. **Дефолт — один глобальный.** Каталог засевается текущим `i3j2sm66q8`,
   помеченным как дефолт. Проекты без явного выбора, unassigned-поды и
   admin-поды идут на глобальный дефолт. Поведение не меняется, пока админ
   явно не переключит проект.
4. **Идентичность записи = `template_id`** (он уникален). Отдельные слаги/id не
   нужны. `label` — только для отображения, свободно переименовывается.
5. **CLI-fallback** тоже использует выбранный `template_id` (через
   `--template-id`); имя Docker-образа в каталоге НЕ храним — оно определяется
   шаблоном на стороне RunPod.
6. **Авторетрай** перерезолвит образ из проекта на КАЖДОЙ попытке (не «снимок»
   на момент заявки): если админ сменил образ проекта, пока заявка висит,
   следующая попытка пойдёт уже на новом образе.

## Модель данных

Три новых ключа в `admin_settings.json` (бэкфиллятся из `DEFAULT_SETTINGS`):

```python
"pod_image_catalog": [
    {"label": "Текущий (comfy_runpod)", "template_id": "i3j2sm66q8"}
],
"default_pod_image": "i3j2sm66q8",   # template_id дефолтной записи
"project_pod_image": {},             # {project: template_id}; нет ключа → дефолт
```

- `pod_image_catalog` — список (сохраняет порядок добавления для UI-редактора
  «+/удалить»). Каждая запись: `{"label": str, "template_id": str}`.
- `default_pod_image` — `template_id` дефолтной записи; обязан присутствовать
  в каталоге.
- `project_pod_image` — маппинг `project → template_id` (по аналогии с
  `project_quotas`). Отсутствие ключа = проект на дефолте.

Засев каталога текущим `i3j2sm66q8` гарантирует неизменность поведения до
первого действия админа. `project_pod_image` стартует пустым.

## Резолвер (единый источник истины)

```python
def resolve_template_id(project):
    """template_id для запуска пода проекта. project может быть None
    (unassigned / admin без проекта) — тогда дефолт."""
    s = get_settings()
    valid = {e["template_id"] for e in s.get("pod_image_catalog") or []}
    tid = (s.get("project_pod_image") or {}).get(project)
    if tid in valid:                      # проект выбрал, и запись ещё существует
        return tid
    if s.get("default_pod_image") in valid:
        return s["default_pod_image"]
    return PRESET["template_id"]          # последний рубеж: каталог пуст/сломан
```

Приоритет: явный выбор проекта (если запись ещё в каталоге) → глобальный дефолт
→ `PRESET["template_id"]` как страховка.

## Поток деплоя (правки сигнатур минимальны)

- `create_pod_via_graphql(name, template_id=None)`:
  в `variables.input.templateId` подставляется `template_id or PRESET["template_id"]`.
  Остальной input не меняется.
- `create_pod(name, bypass_window=False, template_id=None)`:
  прокидывает `template_id` в `create_pod_via_graphql`; в CLI-fallback подставляет
  его в `--template-id` (новая CLI) / `--templateId` (старая CLI). Имя образа
  по-прежнему из шаблона.
- `api_pods_post` (`POST /api/pods`):
  `template_id = resolve_template_id(ap)` →
  `create_pod(name, bypass_window=admin, template_id=template_id)`.
  `ap` — проект юзера, либо выбранный админом проект, либо `None` (→ дефолт).
- `process_pending_requests` (авторетрай):
  на каждой попытке `template_id = resolve_template_id(req["assigned_project"])`
  → `create_pod_via_graphql(req["pod_name"], template_id=template_id)`.

## Admin API

### GET `/api/admin/settings`
Добавить в ответ:
- `pod_image_catalog` — как есть из настроек; если пуст, в ответ отдаётся
  каталог с дефолтной записью (засев из `DEFAULT_SETTINGS`);
- `default_pod_image`;
- `project_pod_image` — как есть (БЕЗ бэкфилла). Отсутствие ключа для проекта
  означает «дефолт»; фронт сам итерирует по `PROJECTS` и для непрописанных
  проектов показывает пункт «По умолчанию».

### POST `/api/admin/settings`
Принимать те же три ключа. Валидация (следуя стилю существующих типизированных
проверок):

- **`pod_image_catalog`** (если `isinstance(..., list)`):
  - для каждой записи: `label` — непустая строка после `strip()`, обрезается до
    60 символов; `template_id` — непустая строка после `strip()`, должна
    матчить `^[A-Za-z0-9_-]+$`;
  - дубли по `template_id` отбрасываются (остаётся первая);
  - если после фильтрации список пуст — **сохраняется прежний каталог**
    (каталог не может стать пустым).
- **`default_pod_image`** (если присутствует): должен существовать в (новом)
  каталоге; иначе берётся `template_id` первого элемента каталога.
- **`project_pod_image`** (если `isinstance(..., dict)`):
  - принимаются только ключи из `PROJECTS`;
  - значение должно быть в каталоге; иначе ключ НЕ сохраняется (проект уходит
    на дефолт).

Все три обрабатываются в существующем `api_admin_settings_post` рядом с
`project_quotas`. Сохранение через `save_settings`/общий `s[...] = ...`.

## Admin UI (`FRONTEND_HTML`, секция настроек)

Новый блок «🖼 Образы подов»:

- **Редактор каталога**: строки `[название] [template_id] [⭐дефолт] [🗑]`
  + кнопка «+ Добавить образ». Выбор дефолта — радио/звёздочка (ровно одна
  активна). Дефолтную строку удалить нельзя (🗑 заблокирован, подсказка
  «нельзя удалить дефолтный образ»).
- **Сетка пер-проект** (визуально как сетка квот): на каждый `PROJECTS` —
  `<select>` с понятными названиями из каталога; первый пункт «По умолчанию
  (<label дефолта>)» соответствует отсутствию ключа в `project_pod_image`.
- Сохранение — в общий `POST /api/admin/settings` вместе с прочими настройками
  (тот же паттерн сборки тела, что у `project_quotas`).

## Крайние случаи и обработка ошибок

- **Удаление образа, на который смотрят проекты**: при сохранении их записи в
  `project_pod_image` отбрасываются валидацией → проекты молча уходят на дефолт.
- **Неверный/«мусорный» `template_id`**: заранее у RunPod не проверяем. Если
  шаблон не существует, мутация вернёт ошибку, которая пойдёт обычным путём
  (тост/лог), как при любой ошибке деплоя сейчас.
- **Визуальная обратная связь**: `imageName` из ответа мутации уже выводится на
  карточке пода — видно, на каком образе реально поднялся под.

## Тестирование

`tests/` (stdlib `unittest`, как `tests/test_pod_request.py`):

1. **Резолвер** `resolve_template_id`:
   - проект с выбором → его `template_id`;
   - проект без выбора → дефолт;
   - проект с выбором, который удалён из каталога → дефолт;
   - `project=None` → дефолт;
   - пустой/сломанный каталог → `PRESET["template_id"]`.
2. **Валидация POST**:
   - дубли `template_id` схлопываются;
   - пустой присланный каталог → прежний сохранён;
   - `default_pod_image` вне каталога → первый элемент;
   - `project_pod_image` с ключом вне `PROJECTS` или значением вне каталога →
     ключ не сохранён.
3. **Бэкфилл GET**: пустые настройки → каталог с дефолтной записью в ответе.
4. **Деплой (мок)**: `create_pod_via_graphql(name, template_id="X")` кладёт
   `"X"` в `variables.input.templateId`; без аргумента → `PRESET["template_id"]`.

## Вне области (YAGNI)

- Не храним имя Docker-образа в каталоге (определяется шаблоном).
- Не валидируем существование шаблона у RunPod заранее.
- Не делаем выбор образа на уровне отдельного юзера или на момент `+New Pod`
  (решено: пер-проект).
- Не трогаем прочие поля `PRESET` (gpu, network volume, ports и т.д.) — образ
  и его шаблон отвечают за окружение.

# Login: убрать автоматический выбор проекта CV

**Дата:** 2026-04-22
**Scope:** экран первичной регистрации юзера (`#loginScreen` в `runpod_manager.py`)

## Проблема

На экране входа юзер вводит nickname и выбирает проект из селекта. Селект рендерится динамически из `/api/projects`, первой опцией оказывается `CV`, и браузер делает её выбранной по умолчанию. Невнимательный юзер жмёт Enter и попадает в `CV`, хотя работает совсем не там. Нужно, чтобы по умолчанию проект был НЕ выбран и юзер был вынужден сделать осознанный выбор.

## Решение

### Frontend (две правки в `runpod_manager.py`)

**1. HTML-плейсхолдер в селекте** (строка ~1883):

Добавить первой опцией disabled+selected плейсхолдер с пустым value:

```html
<select id="loginProj">
  <option value="" disabled selected>— Выберите проект —</option>
</select>
```

Динамически добавляемые опции из `/api/projects` (строка ~2019) встанут после плейсхолдера и не изменят поведения: плейсхолдер останется выбранным, пока юзер не выберет явно.

Плюс disabled-опции: её нельзя выбрать обратно (то есть после выбора реального проекта не вернуться в "не выбран") — это ок, так как это разовый выбор на экране регистрации.

**2. Client-side валидация** в `doUserLogin()` (строка ~2023):

После проверки nickname добавить проверку project:

```js
const n=$('loginNick').value.trim(),p=$('loginProj').value;
if(!n){toast('Enter a nickname','er');return}
if(!p){toast('Выберите проект','er');return}   // ← новая строка
```

Пустое `p` появляется только когда плейсхолдер остался выбран.

### Backend (ничего не меняем)

`validate_registration_input` (`runpod_manager.py:726`) уже отклоняет всё, что не из `PROJECTS`:

```python
if not isinstance(proj, str) or proj not in PROJECTS:
    raise UserValidationError("Invalid or missing project")
```

Пустая строка не в `PROJECTS` → 400. То есть обход через DevTools/curl не прокатит: сервер уже это ловит. Client-side чек нужен чисто для UX (тост вместо сетевого 400).

## Defence in depth — как нельзя случайно залогиниться

| Вектор | Защита |
|---|---|
| Юзер жмёт Enter не выбрав проект | Client-side тост "Выберите проект" |
| Юзер шлёт `POST /api/user/register {project:""}` через DevTools | Server-side `UserValidationError` → 403 |
| Юзер шлёт `project:"FAKE"` | То же самое — `proj not in PROJECTS` |
| Browser autofill/cached session | localStorage хранит только nickname (строка 2016), project не восстанавливается из кэша на логин-экране |
| Юзер уже залогинен | `refreshSession()` скрывает `#loginScreen` до того, как мы дойдём до селекта — наш плейсхолдер не мешает |

## Что НЕ меняем

- Текст заголовка "Enter your name and select your project" — остаётся как есть (вне scope).
- Серверную валидацию — уже корректна.
- Логику восстановления сессии — нет изменений.
- Админский селект `#adminAssignProject` — это отдельный компонент, у него свой legitimate default ("Мой проект (ADMIN)").

## Тесты

Ручная проверка:

1. Очистить cookies + localStorage.
2. Открыть страницу → на экране логина в селекте "— Выберите проект —".
3. Ввести имя, не трогая селект, нажать Enter → тост "Выберите проект", логин не проходит.
4. Выбрать проект (например, DV), Enter → логин проходит, userTag показывает правильный проект.
5. В DevTools отправить `fetch('/api/user/register',{method:'POST',headers:{'Content-Type':'application/json'},body:'{"nickname":"test","project":""}'})` → ответ 403, юзер не зарегистрирован.

Автотесты (`tests/test_migration.py` и пр.) не затрагиваются — фронтендные изменения и серверная проверка уже покрыта существующим валидатором.

## Риски

Минимальные. Два текстовых изменения в одном файле, без миграций и изменений API. Откатывается в одну правку.
